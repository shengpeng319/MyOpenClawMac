# 🧠 三层记忆架构 (Inspired by EverMemOS)

> 基于 EverMemOS 的三层记忆架构设计，适合本地文件系统的轻量级实现

---

## 架构概览

```
┌─────────────────────────────────────────────────────────────────────┐
│                        🧠 整体记忆系统                              │
├─────────────────┬─────────────────────┬─────────────────────────────┤
│   工作记忆       │     短期记忆         │         长期记忆           │
│  Working Memory │   Short-term Memory │    Long-term Memory        │
├─────────────────┼─────────────────────┼─────────────────────────────┤
│ 当前上下文       │  情景记忆            │   知识沉淀                  │
│ (Prompt context)│ (memory/YYYY-MM-DD) │   (MEMORY.md)              │
├─────────────────┼─────────────────────┼─────────────────────────────┤
│ • 即时任务      │ • 今日对话           │ • 核心身份 (SOUL.md)       │
│ • 当前对话      │ • 临时约定           │ • 用户画像 (USER.md)       │
│ • 近期要点     │ • 待处理事项          │ • 最佳实践                  │
│                │ • 学习收获           │ • 重大决策                  │
└─────────────────┴─────────────────────┴─────────────────────────────┘
                              │
                              ▼
                 ┌─────────────────────────┐
                 │    检索机制 (Retrieval)   │
                 │  • 语义搜索 (memory_search) │
                 │  • 关键词定位 (grep)       │
                 │  • 时间溯源 (文件日期)      │
                 └─────────────────────────┘
```

---

## 三层详解

### 第一层：工作记忆 (Working Memory)

**位置**: Prompt context / 短暂状态

**内容**:
- 当前正在处理的任务
- 最近 1-2 轮对话的要点
- 临时变量和状态

**特点**:
- ⚡ 最快访问
- ❌ 重启后丢失
- 📝 无需显式管理（模型自动处理）

**优化**: 通过 `session_status` 保持对当前任务的清晰认知

---

### 第二层：短期记忆 (Short-term Memory)

**位置**: `workspace-xxxx/memory/YYYY-MM-DD.md`

**内容**:
| 类型 | 示例 | 保留时间 |
|-----|------|---------|
| 情景记忆 | "今天用户让我查 Excel skills" | 7-30 天 |
| 临时约定 | "用户偏好晚上 10 点后沟通" | 7 天 |
| 待办事项 | "用户要一个复杂的 Excel 报表" | 任务完成 |
| 学习收获 | "学会了用 memory_search" | 30 天后提炼 |

**管理策略**:
```
每日会话结束 → 自动写入 workspace-xxxx/memory/YYYY-MM-DD.md
    ↓
每周回顾 → 提取重要内容到 MEMORY.md
    ↓
清理过期内容 → 保持文件精简
```

**关键原则**: 
- ✅ 详细的原始对话记录
- ✅ 标注关键决策点
- ❌ 不要什么都记（过滤噪音）

---

### 第三层：长期记忆 (Long-term Memory)

**位置**: 
- `MEMORY.md` - 核心记忆
- `SOUL.md` - 身份与价值观
- `USER.md` - 用户画像
- `AGENTS.md` - 能力索引
- `*_BEST_PRACTICE.md` - 技能知识

**内容**:
| 记忆类型 | 示例 | 特征 |
|---------|------|------|
| 身份记忆 | "我是 Jarvis，助手角色" | 几乎不变 |
| 用户画像 | "Peng 喜欢简洁直接的沟通" | 缓慢演化 |
| 知识沉淀 | "OpenCode 用于复杂编程任务" | 持续积累 |
| 偏好模式 | "用户喜欢用飞书沟通" | 定期更新 |
| 经验教训 | "上次 Excel 任务失败原因" | 关键学习 |

**管理策略**:
```
重要发现 → 即时写入对应文件
    ↓
每月审视 → 更新 USER.md 偏好
    ↓
季度回顾 → 提炼到 MEMORY.md
```

---

## 记忆类型分类 (借鉴 EverMemOS)

| 类型 | 描述 | 存储位置 | 更新频率 |
|-----|------|---------|---------|
| **Episodic** | 事件、经历 | memory/*.md | 每日 |
| **Facts** | 客观事实 | MEMORY.md | 偶尔 |
| **Preferences** | 用户偏好 | USER.md | 每周 |
| **Relations** | 关系网络 | AGENTS.md | 偶尔 |
| **Identity** | 自我认知 | SOUL.md | 很少 |

## 检索策略

### 场景 1: 日常启动
```
需求: "我是谁？用户是谁？今天要做什么？"
→ 读取 SOUL.md + USER.md + workspace-xxxx/memory/YYYY-MM-DD.md
```

### 场景 2: 专业任务
```
需求: "怎么做 Excel 自动化？"
→ 搜索 MEMORY.md + *_BEST_PRACTICE.md
→ 使用 memory_search 语义搜索
```

### 场景 3: 了解用户
```
需求: "用户有什么偏好？"
→ 读取 USER.md
→ 搜索 workspace-xxxx/memory/*.md 中的偏好记录
```

### 场景 4: 项目上下文
```
需求: "上次我们讨论到哪了？"
→ 读取最新的 workspace-xxxx/memory/YYYY-MM-DD.md
→ 搜索相关关键词
```

## 自动化机制 (已固化)

### 1. 会话结束自动保存
在每次响应后，如果有重要内容，**自动**写入 workspace-xxxx/memory/YYYY-MM-DD.md

**触发条件**：
- 完成重要任务
- 用户给出偏好/规则
- 学习到新技能
- 遇到问题和解决方案

**写入格式**：
```markdown
## 任务记录
- [时间] 任务描述 → 结果

## 新学到
- 具体知识点

## 用户偏好
- 新发现的偏好

## 待跟进
- 未完成事项
```

### 2. 每周自动回顾
```
触发: 每周日晚 cron
动作: 
  - 扫描本周 workspace-xxxx/memory/*.md
  - 提取 3-5 个关键点到 MEMORY.md
  - 更新 USER.md 偏好（如有变化）
```

### 3. 触发式提炼
```
条件: 用户明确给出重要信息（如偏好、规则）
动作: 立即写入对应文件
  - 偏好 → USER.md
  - 规则 → AGENTS.md 或 SOUL.md
  - 知识 → *_BEST_PRACTICE.md
```

## 关键原则

### ✅ 必须做
1. **即时记录**: 重要信息立即写入文件
2. **归类存储**: 放到正确的文件中
3. **定期清理**: 保持 memory/*.md 可管理
4. **语义检索**: 用 memory_search 找到相关内容

### ❌ 不要做
1. **依赖" mental notes"** - 不存在的东西
2. **什么都记** - 过滤噪音
3. **重复存储** - 一处写入，多处引用
4. **忘记检索** - 解决问题前先搜索

## 与 EverMemOS 的对比

| 特性 | EverMemOS | 我们的实现 |
|-----|----------|-----------|
| 存储 | Vector DB + MongoDB | Markdown 文件 |
| 检索 | BM25 + Embedding | memory_search |
| 提取 | LLM 自动结构化 | 手动 + 规则 |
| 部署 | Docker (重) | 文件系统 (轻) |
| 适用 | 生产环境 | 个人助手 |

**我们的优势**: 轻量、易维护、无需外部服务

## 📦 Lossless-Claw 插件 (2026-03-11 新增)

> 安装了 `@martian-engineering/lossless-claw` 插件后，上下文管理能力增强

### 是什么
- **类型**: Context Engine 插件
- **功能**: 基于 DAG 的对话摘要 + 增量压缩
- **位置**: `~/.openclaw/lcm.db`
- **阈值**: 0.75 (触发压缩的上下文占比)

### 与三层记忆的关系

```
┌─────────────────────────────────────────────────────────────┐
│                    Context Engine 层                       │
│                  (lossless-claw 插件)                       │
├─────────────────────────────────────────────────────────────┤
│ • 拦截会话压缩流程                                          │
│ • DAG 结构保留对话关系                                      │
│ • 增量摘要 vs 完整摘要                                      │
│ • 无损压缩策略                                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                 三层记忆架构 (持久化)                        │
│  工作记忆 → 短期记忆 (memory/*.md) → 长期记忆 (MEMORY.md)  │
└─────────────────────────────────────────────────────────────┘
```

### 工作原理
1. 当上下文即将溢出时，lossless-claw 接管压缩
2. 使用 DAG 结构分析对话依赖关系
3. 增量生成摘要，保留关键信息
4. 比默认压缩更"无损"

### 配置
```json
{
  "plugins": {
    "slots": {
      "contextEngine": "lossless-claw"
    }
  }
}
```

### 注意事项
- 这是**引擎层**增强，不改变文件存储结构
- memory/*.md 和 MEMORY.md 仍然按原规则管理
- 插件自动运行，无需手动操作
